项目名称：
LUI-for-All

项目目标：
把现有前后端项目从 GUI-first 适配一层独立的 language-first。
用户优先通过自然语言完成任务。
系统优先返回自然语言结果。
只有在文本不足以表达时，才返回临时声明式 UI 组件。
系统对用户项目代码只读。
系统只能在自己的工作目录内写文件。
系统默认只执行只读请求。
所有非只读操作必须经过策略检查和人工确认。

一、固定技术栈

1. 后端主栈
- Python 3.12 作为运行基线，兼容 3.13
- FastAPI 作为主 API 服务框架
- Pydantic v2 作为所有输入输出、配置、协议、结构化模型的唯一类型系统
- LangGraph 作为任务编排、状态机、事件流、人工确认中断、恢复执行的唯一编排框架
- httpx 作为所有 HTTP 请求执行客户端
- SQLAlchemy 2 作为 ORM 和数据库访问层
- Alembic 作为数据库迁移工具
- PostgreSQL 18 作为主数据库
- Redis 8 作为缓存、短期会话、幂等键、LangGraph checkpoint 辅助存储
- OpenTelemetry SDK + OTLP Exporter + Collector 作为 tracing / metrics / logs 的统一遥测层
- uv 作为 Python 包管理和虚拟环境工具

2. 前端主栈
- Vue 3 + TypeScript
- Vite 8
- Vue Router 4
- Pinia 3
- Element Plus 作为基础组件库
- Apache ECharts 6 作为图表渲染库
- pnpm 10 作为前端包管理器

3. 测试与质量
- Pytest 作为后端单元测试和集成测试框架
- Vitest 作为前端单元测试框架
- Playwright 作为端到端测试框架
- Ruff 作为 Python lint + format
- ESLint + Prettier 作为前端 lint + format

4. 解析与扫描
- Tree-sitter 作为代码级静态解析引擎
- 不使用 LLM 直接全仓盲读源码做路由发现
- 必须先做确定性提取，再做 LLM 补全和归纳

5. 制品与对象存储
- 开发环境使用 MinIO
- 生产环境使用 S3 兼容对象存储
- 用于存放原始扫描产物、路由图版本、能力图版本、审计快照、导出图表、请求回放附件

二、架构总原则

1. 不做成 API Chat
本项目不是“对着 Swagger 聊天”。
本项目是“把现有系统改造成 language-first 的交互中间层”。

2. 不让模型直接写前端页面
模型只能输出声明式 UI Block JSON。
模型不能直接输出 Vue 组件源码、HTML 页面或可执行前端脚本。

3. 不让模型直接碰用户项目
用户项目目录只读挂载。
系统只在 .lui_workspace 下写中间文件、缓存、扫描结果和生成物。

4. 不让模型直接发危险请求
所有请求必须经过：
Task Planning -> Policy Check -> Optional Dry Run -> Optional Human Approval -> Execute -> Summarize -> Audit Log

5. 先建模，后执行
必须先生成 Route Map 和 Capability Graph。
没有建模完成，不允许进入真实请求执行阶段。

2. Route Discovery Engine
必须采用插件式设计。

第一批扫描器：
- FastAPI scanner
- Express scanner
- NestJS scanner
- Spring Boot scanner
- Frontend API usage scanner
- OpenAPI ingestor

FastAPI 扫描规则：
- 识别 @app.get/post/put/delete/patch/head/options
- 识别 APIRouter
- 识别 include_router
- 提取 path、method、handler、tags、dependencies、response_model、security scopes
- 若可运行并存在 /openapi.json，则优先读 openapi

Express 扫描规则：
- 识别 app.get/post/put/delete/patch/use
- 识别 router.get/post/put/delete/patch/use
- 识别 express.Router()
- 提取 middleware chain
- 提取 path params
- 提取 handler name
- 提取 auth middleware 命名线索

NestJS 扫描规则：
- 识别 @Controller
- 识别 @Get/@Post/@Put/@Delete/@Patch
- 识别 DTO、Guards、Roles、Interceptors、Swagger decorators
- 若项目使用 @nestjs/swagger，则优先走 OpenAPI intake

Spring Boot 扫描规则：
- 识别 @RestController
- 识别 @RequestMapping/@GetMapping/@PostMapping/@PutMapping/@DeleteMapping/@PatchMapping
- 提取 validation 和 security annotations
- 若存在 springdoc-openapi，则优先走 OpenAPI intake

Frontend API Usage Scanner：
- 扫描 axios.create / request.ts / api.ts / fetch wrapper
- 标记哪些接口被现有 GUI 真正使用
- 输出 used_by_gui / declared_only / orphan_route 三类标签

五、AI 建模原则

1. 先确定性提取，再 LLM 补全
- 路由存在性必须来自确定性证据
- LLM 只能补充解释、归类、合并、命名、生成意图示例
- LLM 不能凭空发明不存在的路由或权限

2. 证据优先级
- openapi/官方文档导出 > 代码 AST 提取 > 前端调用痕迹 > 命名推断 > LLM 语义推断

3. LLM 参与的 5 个任务
- 路由说明归纳
- capability 聚合
- 用户意图匹配
- 结果总结
- UI block 生成

4. LLM 不参与的 5 个任务
- 路由存在性判断
- HTTP method 识别
- 原始权限证据抽取
- 原始风险基线评分
- 执行权限最终放行

六、权限分级与安全分级

1. 权限等级 permission_level
- public
- authenticated
- operator
- admin
- system

自动权限分级信号来源：
- FastAPI dependencies / security scopes / OAuth2 依赖
- Express middleware 名称和链路
- NestJS guards / roles decorators
- Spring Security annotations
- route path 前缀
- handler 名称
- DTO / schema 命名
- 前端页面入口位置（后续版本）

规则：
- 先做确定性规则分类
- LLM 只能上调权限等级，不能无证据下调
- 必须记录分级原因 reasons

2. 安全等级 safety_level
- readonly_safe
- readonly_sensitive
- soft_write
- hard_write
- critical

自动安全分级信号来源：
- HTTP method
- path token
- handler token
- DTO token
- 是否调用外部系统
- 是否涉及财务、权限、消息发送、库存、用户状态
- 是否不可逆
- 是否可能造成生产副作用

默认规则：
- GET/HEAD 且无敏感数据 -> readonly_safe
- GET/HEAD 但返回敏感数据 -> readonly_sensitive
- POST 但语义为查询、导出预览、统计计算 -> readonly_sensitive 或 soft_write
- create/update/approve/refund/pay/send/sync/delete -> hard_write 或 critical
- 权限修改、资金变更、消息发送、脚本执行、生产环境运维 -> critical

3. 执行矩阵
- readonly_safe：自动执行
- readonly_sensitive：绑定用户身份 + 脱敏后自动执行
- soft_write：必须 dry run + 明确确认
- hard_write：V1 默认禁用，保留审批链但不执行
- critical：V1 直接阻断

4. 默认红线
- 不允许自动删除
- 不允许自动支付
- 不允许自动发消息给真实用户
- 不允许自动修改权限
- 不允许自动写数据库
- 不允许自动执行 shell / migration / script 到目标项目

七、执行器 Executor

1. 执行器职责
- 根据 Task Plan 发起真实 HTTP 请求
- 不允许绕过 policy engine
- 不允许由模型自行拼接任意 URL
- 所有目标 host 必须在 allowlist

2. 执行器要求
- 基于 httpx
- 支持 request timeout
- 支持 retry 但仅限幂等请求
- 支持 trace_id 注入
- 支持请求/响应双重日志
- 支持原始响应和脱敏响应分离存储

3. 执行器安全要求
- target project 只读挂载
- secrets 通过环境变量或 secret manager 注入
- executor 无权修改用户项目
- executor 不得访问 allowlist 之外的域名
- executor 不得执行本地 shell 到用户项目目录

十七、实现约束

- 所有 Pydantic schema 必须可序列化
- 所有 LangGraph 节点输入输出必须有类型
- 所有 UI block 必须走白名单 schema 校验
- 所有 policy verdict 必须有 reasons
- 所有 model output 必须结构化校验
- 所有外部调用必须打 trace_id
- 所有任务必须有 task_run_id
- 所有日志必须能追溯到 session_id 和 project_model_version

这份栈是按你现有能力收束后的确定版，而不是再发明一整套新体系：FastAPI 仍围绕 Pydantic v2 演进并原生提供 OpenAPI 文档入口；LangGraph 当前仍把 durable execution、streaming、human-in-the-loop 作为核心能力；Vue 当前稳定版是 3.5.30，Vite 8 已发布，Pinia v3 和 Vue Router 4 仍是官方主线组合。

声明式 UI 和事件层也不需要重复造轮子：A2UI 当前提供 stable v0.8 规范并强调 declarative、安全和预批准组件目录；AG-UI 官方列出了对 LangGraph、A2A 和 MCP Apps 的支持；OpenTelemetry 仍是供应商中立的 traces/metrics/logs 标准；PostgreSQL 当前最新大版本是 18，Redis Open Source 8 已取代 Redis Stack；ECharts 6 已正式发布。

项目建模层之所以要求“先 OpenAPI intake，再静态扫描，再前端使用痕迹交叉”，是因为 FastAPI、NestJS、springdoc 都能在不少项目里直接导出或运行时生成 OpenAPI，而 Express 的路由模式与 Tree-sitter 的增量语法树又很适合做代码级路由发现。
